# API 网关常见故障排查

## 偶发 5xx / 接口偶发失败

### 现象
同一接口大部分请求正常，少量请求返回 502/503/504，无规律、重试常能成功。

### 排查步骤
1. 用响应头里的 `Hi-Trace-Id` 去链路平台查这条失败请求的完整链路，定位是网关→上游哪一跳失败。
2. 查网关对该上游的**健康检查**记录：`GET /admin/upstreams/<name>/health`。偶发失败时常见到某个后端实例**健康状态在 healthy/unhealthy 之间抖动**（flapping）。
3. 看网关访问日志里失败请求的 `upstream_addr` 与 `upstream_status`：若集中在某一两个后端 IP，说明是**特定实例**的问题，而非网关本身。
4. 确认网关到上游的超时配置：`proxy_read_timeout`（默认 60s）；502 多为连接被拒/重置，504 多为上游超时未响应。

### 常见原因
- **上游实例偶发不健康**：后端实例被健康检查间歇性判定为 unhealthy 后摘除、恢复后又加回，期间命中它的请求失败。这种"实例抖动"几乎总是**上游应用自身的问题**（如重启、GC 停顿、被平台杀掉），网关只是如实反映后端状态。
- 上游连接池打满、后端处理变慢导致排队超时。
- 网关与上游之间网络抖动（联系 SRE 查交换机）。

> ⚠️ 排查提示：网关侧看到"上游实例偶发 unhealthy / 频繁摘除恢复"时，**根因通常不在网关**，要去对应**应用所在的容器/运行平台**确认实例为什么会不健康（重启？OOM 被杀？探针失败？）。网关文档只覆盖到"上游状态"这一层。

## 健康检查与摘流

网关按 `health_check` 配置周期探测每个上游实例；连续失败达阈值即摘除（停止转发），恢复后自动加回。摘除/恢复事件记录在网关事件日志：`grep upstream_health /var/log/gateway/event.log`。

- 查当前上游健康状态：`GET /admin/upstreams/<name>/health`
- 查某实例最近的摘除/恢复时间线：`grep <upstream_ip> /var/log/gateway/event.log | tail -50`
